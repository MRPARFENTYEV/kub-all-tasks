# Домашнее задание к занятию «Компоненты Kubernetes»

Среда проекта: production-кластер Kubernetes. Приложение упаковано в **Helm-чарт** для деплоя в разные окружения (dev/stage/prod) — на расчёт ресурсов влияет только количество реплик в values, не сам Helm.

---

## 1. Расчёт ресурсов приложения

| Компонент | Копий | RAM на копию | CPU на копию | RAM итого | CPU итого |
|-----------|-------|--------------|--------------|-----------|-----------|
| База данных | 3 | 4 ГБ | 1 | 12 ГБ | 3 |
| Кеш | 3 | 4 ГБ | 1 | 12 ГБ | 3 |
| Frontend | 5 | 50 МБ | 0,2 | 0,25 ГБ | 1 |
| Backend | 10 | 600 МБ | 1 | 6 ГБ | 10 |
| **Итого** | **21 Pod** | | | **≈ 30,25 ГБ** | **≈ 17 CPU** |

Расшифровка:
- Frontend: 5 × 50 МБ = 250 МБ ≈ **0,25 ГБ**; 5 × 0,2 = **1 CPU**
- Backend: 10 × 600 МБ = 6000 МБ ≈ **6 ГБ**; 10 × 1 = **10 CPU**

**Итого для рабочей нагрузки:** ~**30,25 ГБ RAM** и ~**17 vCPU**.

---

## 2. Служебные ресурсы на нодах

Kubernetes и ОС резервируют часть ресурсов на каждой ноде — Pod'ы не могут использовать 100%.

| Тип ноды | Назначение | RAM (система) | CPU (система) |
|----------|------------|---------------|---------------|
| **Worker** | kubelet, kube-proxy, CNI, container runtime, OS | ~2 ГБ | ~1 vCPU |
| **Control plane** | API server, etcd, scheduler, controller-manager | ~1,5 ГБ | ~0,5 vCPU |

> Для worker-нод используем [рекомендации Kubernetes](https://kubernetes.io/docs/setup/best-practices/cluster-large/) по резервированию ресурсов под системные компоненты.

---

## 3. Выбор размера и количества worker-нод

### Требования к размещению

- DB и Cache по **3 реплики** с отказоустойчивостью → реплики должны быть на **разных нодах** (минимум **3 worker-ноды**).
- Backend (10 реплик) и Frontend (5 реплик) распределяются по оставшимся ресурсам.

### Выбранный размер одной worker-ноды

| Параметр | Значение |
|----------|----------|
| vCPU | **8** |
| RAM | **16 ГБ** |
| Allocatable (после резерва системы) | **~7 vCPU**, **~14 ГБ RAM** |

### Расчёт количества worker-нод

| | RAM | CPU |
|---|-----|-----|
| Нагрузка приложения | 30,25 ГБ | 17 |
| 4 worker × allocatable | 56 ГБ | 28 |
| **После выхода 1 ноды** (3 worker) | **42 ГБ** | **21** |
| Запас после отказа 1 ноды | +11,75 ГБ (~39%) | +4 vCPU (~24%) |

**3 worker-ноды недостаточно:** при отказе одной остаётся 28 ГБ / 21 CPU — на пределе по CPU и без запаса для DB/Cache anti-affinity.

**Вывод: 4 worker-ноды** — при падении одной три оставшиеся (**42 ГБ / 21 CPU**) покрывают нагрузку **30,25 ГБ / 17 CPU** с запасом.

### Пример распределения Pod'ов по 4 worker-нодам

| Worker | Pod'ы | RAM | CPU |
|--------|-------|-----|-----|
| node-1 | DB-1, Cache-1, Backend ×2, Frontend ×1 | ~13,1 ГБ | ~3,2 |
| node-2 | DB-2, Cache-2, Backend ×3, Frontend ×2 | ~13,7 ГБ | ~4,4 |
| node-3 | DB-3, Cache-3, Backend ×3, Frontend ×1 | ~13,1 ГБ | ~3,2 |
| node-4 | Backend ×2, Frontend ×1 | ~2,25 ГБ | ~2,2 |

---

## 4. Control plane (master-ноды)

Для отказоустойчивого кластера — **3 control plane ноды** (etcd quorum, HA API server):

| Параметр | Значение |
|----------|----------|
| Количество | **3** |
| vCPU | **2** на ноду |
| RAM | **4 ГБ** на ноду |

Control plane не запускает рабочие Pod'ы приложения — отдельный пул.

---

## 5. Итоговая конфигурация кластера

| Роль | Кол-во | vCPU | RAM | Примечание |
|------|--------|------|-----|------------|
| **Control plane** | 3 | 2 | 4 ГБ | API, etcd, scheduler |
| **Worker** | 4 | 8 | 16 ГБ | DB, Cache, Frontend, Backend |
| **Итого нод** | **7** | | | |

### Сводка по ресурсам

| | RAM | CPU |
|---|-----|-----|
| Нагрузка приложения | 30,25 ГБ | 17 |
| Worker allocatable (4 ноды) | 56 ГБ | 28 |
| Worker allocatable при отказе 1 ноды | 42 ГБ | 21 |
| Запас при отказе 1 worker | ~39% RAM | ~24% CPU |

## Вывод

Для проекта с Helm-чартом, отказоустойчивой БД (3×4 ГБ), кешем (3×4 ГБ), frontend (5×50 МБ) и backend (10×600 МБ) достаточно:

- **4 worker-ноды**: 8 vCPU, 16 ГБ RAM — с запасом на отказ одной ноды
- **3 control plane ноды**: 2 vCPU, 4 ГБ RAM — для HA control plane

Общий размер кластера: **7 нод**.
